# 软件开发工具实践 - 实验报告合集

## 实验一：Web开发环境搭建与静态站点构建

### 1、实验目的

本次实验旨在安装并运行Web开发环境（Node.js、Git、Hexo），通过搭建一个本地静态博客站点，理解静态站点生成器的工作原理，掌握Web开发环境的基础安装与配置方法。

### 2、实验环境

- 操作系统：Windows 11
- Node.js：v24.18.0（LTS版本）
- Git：v2.55.0.2
- Hexo CLI：v4.3.2
- 代码编辑器：Visual Studio Code
- 浏览器：Microsoft Edge

### 3、实验过程

**步骤一：安装Node.js与npm**

访问Node.js官网（或国内镜像站），下载LTS版本的Windows安装包（node-v24.18.0-x64.msi）。双击安装包，在安装过程中务必勾选"Add to PATH"，其余保持默认。打开命令提示符（CMD），输入`node -v`和`npm -v`验证安装是否成功。配置npm国内镜像源：`npm config set registry https://registry.npmmirror.com`。

**步骤二：安装Git版本管理工具**

访问Git官网下载Windows版本安装包。关键配置项如下：
- PATH环境：选择"Git from the command line and also from 3rd-party software"
- 默认编辑器：选择"Use Visual Studio Code as Git's default editor"
- 行尾转换：保持默认"Checkout Windows-style, commit Unix-style line endings"
- 初始分支名：选择"Override the default branch name"，输入`main`

安装完成后验证：`git --version`，并配置用户信息：`git config --global user.name "你的姓名"`和`git config --global user.email "你的邮箱@qq.com"`。

**步骤三：安装Hexo CLI并初始化博客项目**

全局安装Hexo：`npm install -g hexo-cli`，验证：`hexo -v`。在D盘创建项目目录：`D:`，`mkdir Blog`，`cd Blog`，执行`hexo init my-blog`初始化博客项目。进入项目目录`cd my-blog`，执行`npm install`安装项目依赖。

**步骤四：本地预览博客**

启动本地预览服务：`hexo server`，看到提示`INFO Hexo is running at http://localhost:4000`后，打开浏览器访问`http://localhost:4000`，看到Hexo默认博客首页。

**步骤五：创建并发布一篇博客文章**

使用`hexo new "我的第一篇文章"`创建新文章，用VS Code编辑`source/_posts/我的第一篇文章.md`，修改内容后保存。执行`hexo generate`重新生成静态页面，再次启动预览服务查看效果。

**步骤六：将源码推送到GitHub**

在GitHub创建私有仓库，初始化Git并关联远程仓库，创建`.gitignore`文件忽略`node_modules/`、`public/`、`.deploy_git/`，提交并推送：`git add .`，`git commit -m "初始化Hexo博客项目"`，`git branch -M main`，`git push -u origin main`。

### 4、实验结果

成功搭建了Web开发环境，并在本地运行了Hexo博客站点。访问`http://localhost:4000`可看到默认博客首页，说明Node.js、Git、Hexo均正确安装并配置。源码已成功推送到GitHub远程仓库。

### 5、知识总结

| 工具 | 作用 | 关键命令 |
|------|------|----------|
| Node.js | JavaScript运行时环境 | `node -v`，`npm -v` |
| npm | Node.js包管理器 | `npm install -g`，`npm config set registry` |
| Git | 版本控制工具 | `git init`，`git add`，`git commit`，`git push` |
| Hexo | 静态站点生成器 | `hexo init`，`hexo generate`，`hexo server` |

### 6、出现问题

| 问题现象 | 可能原因 | 解决方案 |
|----------|----------|----------|
| nodejs.org官网打不开 | 海外服务器访问不稳定 | 改用国内镜像站 |
| hexo init克隆超时报错 | GitHub网络连接被重置 | Hexo自动切换为内置缓存复制，等待即可 |
| Git安装后git命令不被识别 | 安装时未勾选"Add to PATH" | 重新运行安装程序，确保勾选"Add to PATH" |
| git push提示认证失败 | 未配置Personal Access Token | 在GitHub生成Token，替代密码使用 |

### 7、心得体会

通过本次实验，我深刻体会到了开源工具的便利性和强大功能。Hexo、GitHub、VS Code等开源工具让我能够零成本搭建个人博客，这体现了开源社区的共享精神。同时，我也认识到国内虽然有Gitee、Coding等平台，但在生态建设上与GitHub仍有差距。


## 实验二：代码编辑与管理（VS Code + Git + Markdown）

### 1、实验目的

本次实验旨在安装Visual Studio Code代码编辑器，掌握Markdown插件的安装与使用，学习Markdown基础语法，并熟悉Git的日常操作流程（add/commit/log/push/pull），理解代码编辑器与集成开发环境的区别。

### 2、实验环境

- 操作系统：Windows 11
- 代码编辑器：Visual Studio Code
- 版本管理工具：Git v2.55.0.2
- Markdown插件：Markdown All in One、Markdown Preview Enhanced
- 浏览器：Microsoft Edge

### 3、实验过程

**步骤一：安装Visual Studio Code**

访问VS Code官网下载Windows版本安装包。在"选择附加任务"页面，务必勾选"添加到PATH"、创建桌面快捷方式、将"通过Code打开"操作添加到Windows资源管理器上下文菜单等选项。

**步骤二：安装Markdown相关插件**

打开扩展面板（Ctrl+Shift+X），安装以下插件：
- Markdown All in One：目录生成、快捷键支持、自动完成
- Markdown Preview Enhanced：数学公式、流程图、导出为PDF/HTML
- Paste Image（可选）：直接粘贴剪贴板中的图片到Markdown文件
- markdownlint（可选）：检查Markdown语法错误

**步骤三：配置编辑器优化写作体验**

开启自动保存（files.autosave设置为afterDelay），开启单词折行（editor.wordwrap设置为on），配置图片粘贴路径为相对路径。

**步骤四：创建并编辑第一个Markdown文件**

新建文件保存为test.md，输入Markdown基础语法内容（标题、列表、代码块等），使用Ctrl+Shift+V打开预览窗口查看渲染效果。

**步骤五：Git版本管理操作**

配置Git用户信息：`git config --global user.name "你的姓名"`和`git config --global user.email "你的邮箱@qq.com"`。将实验报告纳入版本管理：`git add`，`git commit -m "添加实验报告"`，`git push origin main`。查看提交历史：`git log --oneline`。

### 4、实验结果

成功完成了VS Code的安装与配置，Markdown插件安装成功并可正常编辑预览，Git用户信息配置成功，实验报告已提交到GitHub仓库，可通过`git log`查看提交历史。

### 5、知识总结

| 工具/概念 | 作用 |
|-----------|------|
| VS Code | 轻量级代码编辑器，通过插件扩展功能 |
| Markdown | 轻量级标记语言，用于结构化文档编写 |
| Git | 分布式版本控制系统 |
| 插件（Extension） | VS Code的核心扩展机制 |

**Markdown常用语法：**

| 效果 | 语法 |
|------|------|
| 标题 | `# 一级标题`、`## 二级标题` |
| 代码块 | \`\`\`bash ... \`\`\` |
| 表格 | `\| 列1 \| 列2 \|` |
| 加粗 | `**加粗文字**` |
| 列表 | `- 项目1` |

### 6、出现问题

| 问题现象 | 可能原因 | 解决方案 |
|----------|----------|----------|
| VS Code无法识别git命令 | 安装时未勾选"添加到PATH" | 重新运行安装程序，勾选"添加到PATH"；或重启VS Code |
| 扩展面板搜索不到插件 | 网络问题或搜索关键词不准确 | 检查网络连接；尝试使用准确的插件全名搜索 |
| 预览窗口不显示或显示异常 | 插件未正确安装或未启用 | 检查扩展面板中插件是否已启用；重启VS Code |

### 7、心得体会

通过本次实验，我深刻认识到工具的选择和配置对开发效率的重要影响。VS Code作为轻量级编辑器，通过插件机制可以灵活扩展功能，满足不同开发场景的需求。Markdown语法简单易学，适合编写技术文档和实验报告。Git作为版本控制工具，是现代软件开发协作的基础，掌握Git的基本操作是团队协作的前提。


## 实验三：虚拟机安装与配置

### 1、实验目的

本次实验旨在使用虚拟机工具（VirtualBox）安装Linux系统（Ubuntu Server 22.04 LTS），掌握虚拟机的创建、配置和网络设置方法，理解虚拟机网络的三种模式（NAT、桥接、仅主机）及其适用场景。

### 2、实验环境

- 宿主机操作系统：Windows 11
- 虚拟化平台：Oracle VirtualBox 7.0.26
- 客户机操作系统：Ubuntu Server 22.04.5 LTS
- 虚拟机内存：2048 MB
- 虚拟机硬盘：20 GB（动态分配）
- 虚拟机网络：NAT + 端口转发
- SSH工具：OpenSSH Server

### 3、实验过程

**步骤一：安装VirtualBox**

访问VirtualBox官网下载Windows版本安装包，右键以管理员身份运行。在"自定义安装"页面保持所有选项默认勾选，安装过程中如弹出"网络接口警告"，点击"是"继续。

**步骤二：下载Ubuntu Server ISO镜像**

访问Ubuntu官网下载22.04.5 LTS版本，或使用国内镜像站（清华镜像）下载`ubuntu-22.04.5-live-server-amd64.iso`文件。

**步骤三：创建虚拟机**

在VirtualBox主界面点击"新建"，填写虚拟机信息：名称`ubuntu-server`，文件夹放在非系统盘，ISO映像选择已下载的镜像，类型Linux，版本Ubuntu (64-bit)。硬件配置：内存2048 MB，处理器1 CPU，不勾选"启用EFI"。虚拟硬盘：20 GB，不勾选"预先分配全部空间"。

**步骤四：安装Ubuntu Server系统**

启动虚拟机，选择语言English。安装程序更新提示选择"Continue without updating"。安装类型选择"Ubuntu Server"。网络配置保持默认DHCP设置。代理配置留空。镜像源配置系统自动选择清华镜像源。磁盘分区选择"Use an entire disk"，勾选"Set up this disk as an LVM group"，不勾选加密选项。

用户信息配置：Your name为`GroupWork`，Your server's name为`ubuntu-server`，Pick a username为`groupwork`，密码设为`123456`。SSH配置务必按空格键选中`Install OpenSSH server`，保持`Allow password authentication over SSH`为勾选状态。Ubuntu Pro升级选择"Skip for now"。Snap软件包全部不选。等待安装完成后选择"Reboot Now"重启。

**步骤五：登录系统并配置网络**

重启后输入用户名`groupwork`，密码`123456`登录。配置DNS：`echo "nameserver 114.114.114.114" | sudo tee /etc/resolv.conf`和`echo "nameserver 8.8.8.8" | sudo tee -a /etc/resolv.conf`。更换软件源为阿里云镜像，执行`sudo apt update`。

**步骤六：创建小组用户账号**

为每位组员创建独立的SSH登录账号：`sudo adduser cn`，`sudo adduser hcy`，`sudo adduser xrh`，`sudo adduser wyj`。每个用户设置密码为`123456`，其余信息直接回车跳过。验证用户创建成功：`cat /etc/passwd | grep -E 'cn|hcy|xrh|wyj'`。

**步骤七：配置端口转发**

关闭虚拟机，在VirtualBox中设置端口转发：名称`web`，协议`TCP`，主机端口`8080`，子系统端口`80`。重新启动虚拟机，查看宿主机IP地址（`ipconfig`），将此IP和端口号`8080`发给小组成员。

### 4、实验结果

成功完成了VirtualBox的安装、Ubuntu Server 22.04 LTS的系统部署、网络配置、用户管理、端口转发配置。虚拟机NAT模式下可正常上网（`apt update`成功），小组用户账号创建完成（4人），SSH服务正常运行，端口转发配置完成。

### 5、知识总结

| 网络模式 | 特点 | 适用场景 |
|----------|------|----------|
| NAT | 虚拟机通过宿主机上网，外部不可直达 | 日常上网、安装软件 |
| 桥接 | 虚拟机直接接入宿主机所在局域网 | 需要被其他设备直接访问 |
| 仅主机 | 宿主机与虚拟机形成隔离私网 | 搭建内网互通实验环境 |

**关键配置文件：**

| 文件 | 作用 |
|------|------|
| `/etc/resolv.conf` | DNS解析配置 |
| `/etc/apt/sources.list` | 软件源配置 |
| `/etc/passwd` | 用户账户信息 |
| `/etc/group` | 用户组信息 |

### 6、出现问题

| 问题现象 | 可能原因 | 解决方案 |
|----------|----------|----------|
| apt update报Temporary failure resolving | DNS解析失败 | 手动配置DNS：`echo "nameserver 114.114.114.114" \| sudo tee /etc/resolv.conf` |
| apt update卡在0% [Connecting...] | 软件源连接慢或无法访问 | 替换为国内镜像源（阿里云或清华源） |
| 宿主机无法访问10.0.2.15:80 | NAT模式下端口未映射 | 配置端口转发（宿主机8080→虚拟机80） |
| 组员访问宿主机IP:8080超时 | Windows防火墙拦截 | 暂时关闭防火墙测试，或添加入站规则放行8080端口 |

### 7、心得体会

通过本次实验，我深刻理解了虚拟机技术的实用价值。在一台物理计算机上同时运行多个独立的操作系统，为学习和实验提供了极大的便利。虚拟机网络的三种模式各有特点，NAT模式适合日常上网和软件安装，桥接模式适合对外提供服务，仅主机模式适合内网实验。端口转发技术在NAT模式下实现了对外服务的能力，是网络配置中非常实用的技巧。Linux系统的命令行操作虽然初看复杂，但掌握后效率极高，这激发了我深入学习Linux的兴趣。


## 实验四：Web站点构建（Hexo静态博客）

### 1、实验目的

本次实验旨在使用静态站点构建工具（Hexo）构建一个静态Web博客站点，掌握Web应用开发的基本流程（环境准备→项目初始化→内容编写→主题配置→本地预览→部署上线），理解静态站点生成器的工作原理及其与动态站点的区别。

### 2、实验环境

- 操作系统：Windows 11 / Ubuntu Server 22.04
- 静态站点生成器：Hexo v8.1.2
- Node.js：v20.19.0
- 主题：NexT v8.12.1
- 代码编辑器：Visual Studio Code
- 版本控制：Git + GitHub
- Web服务器：Nginx（虚拟机）

### 3、实验过程

**步骤一：安装Hexo CLI**

在宿主机和虚拟机中分别安装Hexo命令行工具：`npm install -g hexo-cli`，验证：`hexo -v`。

**步骤二：初始化博客项目**

在宿主机D盘创建项目目录并初始化：`D:`，`mkdir Blog`，`cd Blog`，`hexo init my-blog`，`cd my-blog`，`npm install`。

**步骤三：本地预览博客**

启动本地预览服务：`hexo server`，访问`http://localhost:4000`看到Hexo默认博客首页。

**步骤四：更换为NexT主题**

克隆NexT主题到themes目录：`git clone https://github.com/theme-next/hexo-theme-next themes/next`。修改站点配置文件`_config.yml`，将`theme`字段改为`next`。启用NexT主题的Tabs标签功能：编辑`themes/next/_config.yml`，将`tabs.enable`改为`true`。配置资源引用方式：在主题配置中设置`vendors: local`。

**步骤五：编写实验报告文章**

使用`hexo new "实验报告标题"`创建新文章，在`source/_posts/`目录下编辑`.md`文件，使用Markdown语法编写实验报告。使用Tabs标签汇总多篇报告，创建`实验报告合集.md`，使用`{% tabs %}`标签将多篇报告整合到一篇文章中。

**步骤六：生成静态页面**

执行`hexo generate`，`public/`目录下生成完整的静态HTML文件。

**步骤七：将站点部署到GitHub**

初始化Git仓库并关联远程仓库，创建`.gitignore`文件，提交并推送：`git add .`，`git commit -m "初始化Hexo博客项目"`，`git branch -M main`，`git push -u origin main`。

**步骤八：部署到虚拟机Nginx服务器**

在虚拟机中拉取代码：`git pull`。安装项目依赖：`npm install`。生成静态文件：`hexo generate`。将静态文件复制到Nginx网站根目录：`sudo cp -r public/* /var/www/html/`。通过端口转发访问站点。

### 4、实验结果

成功完成了Hexo静态博客站点的构建，NexT主题安装并启用成功，Tabs标签功能正常，多篇实验报告通过Tabs汇总，源码推送到GitHub远程仓库，虚拟机拉取并生成静态页面，Nginx正常展示博客页面，组员可通过`宿主机IP:8080`访问。

### 5、知识总结

**静态站点vs动态站点：**

| 对比维度 | 静态站点（SSG） | 动态站点（如WordPress） |
|----------|------------------|--------------------------|
| 页面生成时机 | 构建时预先生成 | 用户访问时实时生成 |
| 数据库 | 不需要 | 通常需要 |
| 访问速度 | 极快 | 较慢 |
| 安全性 | 高 | 需额外防护 |
| 适用场景 | 博客、文档站 | 电商、社交、CMS |

**Web应用开发基本流程：**

环境准备 → 项目初始化 → 内容编写 → 主题配置 → 本地预览 → 部署上线

### 6、出现问题

| 问题现象 | 可能原因 | 解决方案 |
|----------|----------|----------|
| hexo init克隆超时 | GitHub访问不稳定 | 等待Hexo自动切换为内置缓存复制 |
| tabs标签报unknown block tag | 主题未启用Tabs功能 | 安装NexT主题，启用tabs.enable: true |
| NexT主题页面无样式 | 国外CDN资源加载失败 | 设置vendors: local |
| 虚拟机hexo generate报ESM错误 | Node.js版本与Hexo不兼容 | 升级Node.js到20.x，更新Hexo到最新版 |

### 7、心得体会

通过本次实验，我深入理解了静态站点生成器的工作原理。Hexo将Markdown源文件预先编译成完整的HTML静态网页，访问速度极快且安全性高。NexT主题功能丰富，Tabs标签功能为实验报告的汇总展示提供了极大的便利。我认识到选择合适的主题不仅影响站点外观，还决定了可用的功能扩展。版本管理是团队协作的基石，通过Git和GitHub实现高效分工，这是现代软件开发的标准实践。


## 实验五：Linux环境配置与基本操作

### 1、实验目的

本次实验旨在掌握Linux命令行的基本操作（文件操作、目录管理、系统查询），学会使用命令行进行用户与组的管理，理解Linux文件权限模型，掌握环境变量的查看、设置与管理方法，熟悉Linux系统配置文件的编辑与作用范围。

### 2、实验环境

## 实验六：Linux账户与权限管理

### 1、实验目的

本次实验旨在掌握Linux系统用户账户的创建、修改与删除操作，理解用户组的概念并掌握组管理与成员管理，深入理解Linux文件权限模型（rwx权限位与数字表示法），掌握特殊权限（SUID、SGID、Sticky Bit）的作用与配置，理解`sudo`权限的配置方法，掌握系统级与用户级环境变量的配置与管理。

### 2、实验环境

- 操作系统：Ubuntu Server 22.04.5 LTS
- 内核版本：Linux 5.15.0-185-generic
- Shell环境：Bash 5.1.16
- 用户身份：groupwork（具有sudo权限）

### 3、实验过程

**步骤一：查看系统配置文件**

执行`cat /etc/passwd`查看用户账户信息，`sudo cat /etc/shadow`查看加密密码信息，`cat /etc/group`查看用户组信息。理解各文件格式：`/etc/passwd`格式为"用户名:密码占位符:UID:GID:用户描述:主目录:登录Shell"。

**步骤二：用户账户管理**

创建新用户：`sudo adduser devuser`（交互式），设置密码为`123456`。修改用户密码：`sudo passwd devuser`。修改用户属性：`sudo usermod -d /home/newhome devuser`，`sudo usermod -s /bin/zsh devuser`。删除用户：`sudo userdel -r devuser`。

**步骤三：用户组管理**

查看用户所属组：`groups devuser`，`id devuser`。创建用户组：`sudo groupadd devteam`。将用户添加到组：`sudo usermod -aG devteam devuser`。从组中移除用户：`sudo gpasswd -d devuser devteam`。删除用户组：`sudo groupdel devteam`。

**步骤四：文件权限管理**

查看文件权限：`ls -l test.txt`。使用数字法修改权限：`chmod 755 script.sh`，`chmod 644 config.txt`，`chmod 600 secret.txt`。使用符号法修改权限：`chmod u+x file.txt`，`chmod g-w file.txt`，`chmod o+r file.txt`。递归修改目录权限：`chmod -R 755 /path/to/dir`。修改文件所有者和所属组：`sudo chown devuser:devteam file.txt`，`sudo chgrp devteam file.txt`。

**步骤五：特殊权限（SUID、SGID、Sticky Bit）**

SUID设置：`chmod u+s /path/to/program`或`chmod 4755 /path/to/program`。SGID设置：`chmod g+s /path/to/dir`或`chmod 2755 /path/to/dir`。Sticky Bit设置：`chmod +t /path/to/dir`或`chmod 1777 /path/to/dir`。

**步骤六：sudo权限配置**

使用`sudo visudo`编辑`/etc/sudoers`文件，添加配置：`devuser ALL=(ALL) ALL`（需要密码）或`devuser ALL=(ALL) NOPASSWD: ALL`（免密码）。验证sudo权限：`sudo -l`。

**步骤七：环境变量配置**

查看环境变量：`env`，`echo $PATH`，`echo $HOME`。临时设置：`export MY_VAR="Hello Linux"`。永久设置（用户级）：编辑`~/.bashrc`添加自定义配置，执行`source ~/.bashrc`。永久设置（系统级）：在`/etc/profile.d/`创建独立脚本。

### 4、实验结果

成功完成了用户账户管理（创建、修改、删除）、用户组管理（创建、添加成员、移除成员、删除）、文件权限管理（数字法、符号法、递归修改）、特殊权限配置（SUID、SGID、Sticky Bit）、sudo权限配置、环境变量配置（用户级、系统级）等任务。

### 5、知识总结

**特殊权限：**

| 特殊权限 | 作用 | 设置方式 |
|----------|------|----------|
| SUID | 以文件所有者权限执行 | `chmod u+s` 或 `chmod 4xxx` |
| SGID | 以文件所属组权限执行/目录下新文件继承组 | `chmod g+s` 或 `chmod 2xxx` |
| Sticky Bit | 只有文件所有者才能删除 | `chmod +t` 或 `chmod 1xxx` |

### 6、出现问题

| 问题现象 | 可能原因 | 解决方案 |
|----------|----------|----------|
| userdel: user is currently logged in | 用户当前已登录 | 先退出用户或终止进程后再删除 |
| chown: invalid user | 用户名不存在 | 确认用户名正确 |
| visudo保存时报语法错误 | sudoers文件配置格式有误 | 检查语法格式 |
| 设置SUID后显示为S而非s | 文件没有执行权限 | 先用chmod u+x添加执行权限 |

### 7、心得体会

通过本次实验，我深入理解了Linux"一切皆文件"的设计理念。用户、组、权限等核心概念都通过配置文件（`/etc/passwd`、`/etc/group`等）体现。权限模型的多层设计（文件权限、特殊权限、sudo机制）提供了精细的访问控制能力。特殊权限中SUID让普通用户能执行需要更高权限的程序（如passwd修改密码），SGID和Sticky Bit在多用户共享目录时非常实用。这些知识对后续的系统管理和安全运维至关重要。


## 实验七：远程登录管理（SSH）

### 1、实验目的

本次实验旨在掌握OpenSSH服务器的安装与配置方法，理解SSH的两种认证方式（密码认证与密钥认证），掌握SSH密钥对的生成与公钥部署方法实现免密登录，掌握SCP和rsync命令的使用实现跨机文件传输与同步，理解多机开发环境下的远程管理流程。

### 2、实验环境

- 服务器端操作系统：Ubuntu Server 22.04.5 LTS
- 客户端操作系统：Windows 11 / Ubuntu Server
- SSH服务：OpenSSH Server 8.9p1
- SSH客户端：OpenSSH Client（Windows 10/11自带）
- 网络模式：NAT + 端口转发

### 3、实验过程

**步骤一：检查SSH服务状态**

执行`sudo systemctl status ssh`检查服务状态。如未安装，执行`sudo apt install openssh-server -y`。如未运行，执行`sudo systemctl start ssh`和`sudo systemctl enable ssh`。

**步骤二：SSH服务配置文件**

查看配置文件`/etc/ssh/sshd_config`，关键配置项：`Port 22`，`PermitRootLogin no`，`PubkeyAuthentication yes`，`PasswordAuthentication yes`。修改配置后执行`sudo systemctl restart ssh`。

**步骤三：SSH密码认证登录**

从宿主机连接到虚拟机：`ssh groupwork@10.0.2.15`，输入密码`123456`，成功登录后显示`groupwork@ubuntu-server:~$`。

**步骤四：SSH密钥认证配置**

生成SSH密钥对：`ssh-keygen -t ed25519 -C "你的邮箱@qq.com"`。将公钥复制到服务器：`ssh-copy-id groupwork@10.0.2.15`（或手动复制到`~/.ssh/authorized_keys`）。测试免密登录：`ssh groupwork@10.0.2.15`，不再提示输入密码即成功。

**步骤五：SCP命令文件传输**

上传文件：`scp D:\test.txt groupwork@10.0.2.15:/home/groupwork/`。上传目录：`scp -r D:\project groupwork@10.0.2.15:/home/groupwork/`。下载文件：`scp groupwork@10.0.2.15:/home/groupwork/log.txt D:\`。

**步骤六：rsync命令同步**

本地同步：`rsync -avz /home/groupwork/source/ /home/groupwork/backup/`。远程推送：`rsync -avz -e ssh D:\project\ groupwork@10.0.2.15:/home/groupwork/project/`。保持目录完全一致：`rsync -avz --delete -e ssh D:\source\ groupwork@10.0.2.15:/home/groupwork/dest/`。排除特定文件：`rsync -avz --exclude='*.log' --exclude='tmp/' -e ssh D:\source\ groupwork@10.0.2.15:/home/groupwork/dest/`。

### 4、实验结果

成功完成了SSH服务的安装与配置、密码认证登录、密钥对生成与公钥部署、免密登录验证、SCP文件上传/下载、rsync目录同步（含--delete和--exclude）等任务。

### 5、知识总结

**SSH两种认证方式对比：**

| 认证方式 | 原理 | 安全性 | 适用场景 |
|----------|------|--------|----------|
| 密码认证 | 每次登录输入密码 | 较低 | 临时访问 |
| 密钥认证 | 使用公钥/私钥对验证身份 | 高 | 日常管理、自动化脚本 |

**SCP vs rsync：**

| 工具 | 特点 | 适用场景 |
|------|------|----------|
| SCP | 基于SSH的简单复制 | 一次性、少量文件传输 |
| rsync | 增量同步，只传输变化的部分 | 目录同步、定期备份 |

### 6、出现问题

| 问题现象 | 可能原因 | 解决方案 |
|----------|----------|----------|
| Connection refused | SSH服务未运行 | sudo systemctl start ssh |
| Permission denied | 密码错误或密钥未配置 | 确认用户名和密码；检查authorized_keys |
| 免密登录仍需密码 | authorized_keys权限过大 | chmod 600 ~/.ssh/authorized_keys |
| ssh-copy-id不存在 | Windows CMD不支持 | 使用Git Bash执行，或手动复制公钥 |

### 7、心得体会

通过本次实验，我系统学习了SSH远程管理技术。SSH是现代Linux运维的基础工具，无论是单机管理还是多机协作，SSH都是最核心的远程管理协议。密钥认证相比密码认证更安全（防暴力破解）、更便捷（免密登录），是自动化脚本和CI/CD流水线的基础。SCP和rsync各有所长：SCP适合简单单次的小文件传输，rsync适合大目录定期同步（增量传输、带宽高效）。远程管理的安全原则包括禁用root直接登录、使用密钥认证、限制允许登录的用户和IP等，这些都是生产环境的基本要求。
